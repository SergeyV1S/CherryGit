# Доработки до полной готовности CherryGit

Документ составлен на основе функциональных требований ВКР и фактического состояния кодовой базы на 2026-05-22.

## Шкала готовности

- 🟢 **80–100%** — работает, нужны минорные правки
- 🟡 **20–70%** — скелет/частично реализовано
- 🔴 **0–15%** — не реализовано

## 1. Сбор данных из GitLab

### 1.1. GitLab API клиент (FR-01) — 🟢 100%

Готово полностью: PAT auth, пагинация, timeout, retry на 429, маппинг ошибок, шифрование токенов, audit. Доработок нет.

### 1.2. Подключение проектов к командам (UC-01 шаги 6–9) — 🟢 90%

**Реализовано:**

- `connectProject(actorUid, dto)` — UC-01 шаги 6–10 целиком: проверка connection, валидация teamUids, fetch метаданных проекта с GitLab, транзакционный insert в `projects` + `team_projects` + `sync_statuses`
- `updateProject(uid, dto, actorUid)` — патч `releaseTagPattern`/`hotfixLabel`/`revertLabel` + полная замена связей с командами
- `deleteProject(actorUid, uid)` — снятие связей и `sync_statuses`, исторические commits/MR намеренно сохраняются
- `triggerResync(actorUid, uid)` — форсированный пересбор (фактически делегирует в Sync, который ещё `notImplemented`)
- `listProjects()` с агрегацией команд через JOIN
- `getProject(uid)` возвращает `{project, teams, syncStatus}`
- Защита от дубликата через PG unique constraint `uq_project_per_connection`, маппинг `23505` → HTTP 409 (с проверкой и `error.cause.code` на случай оборачивания Drizzle)
- Zod-валидация DTO в контроллере (`connectProjectSchema`, `updateProjectSchema`)
- Audit: `project.connected` / `project.updated` / `project.disconnected` / `project.resync_triggered`
- Endpoint `POST /admin/projects/:uid/resync` добавлен в router
- Fire-and-forget вызов `SyncService.syncProject` при подключении — заработает автоматически когда 1.3 будет готов

**Осталось доработать:**

- **Зависимость от 1.3**: бутстрап-sync и `triggerResync` сейчас возвращают 501 (Sync не реализован). Будет работать после готовности 1.3 без правок в этом модуле.
- **Колонка `projects.defaultBranch`** не добавлена в схему. Для sync (1.3) понадобится либо добавить колонку и миграцию (сохранять `remote.default_branch` при `connectProject`), либо запрашивать `fetchProject(id)` повторно из sync-цикла. Рекомендуется первое — добавить миграцию `projects.default_branch text`.
- **Валидация `releaseTagPattern` как glob** — сейчас сохраняется любая строка. На уровне DTO стоит запретить пустую/пробельную; полная проверка glob нужна только при mатчинге в 1.4.
- **Авторизация владельца GitLab-подключения**: сейчас любой ADMIN может подключить проект к любому connection (формально соответствует ВКР 2.2.7, но если будет «администратор отдела» — потребуется проверка `connection.ownerUid === actorUid OR role === ADMIN`).
- **Пагинация `listProjects`** — для MVP не нужна (десятки проектов), но при росте — добавить limit/offset.

### 1.3. Периодический инкрементальный sync (FR-02) — 🔴 10%

`sync.service.ts` — 34 строки, скелет. **Это самый большой кусок работы.**

**Доработать:**

- **Планировщик**: cron (`node-cron` или setInterval в bootstrap) — обходить все `projects.status='active'` раз в N минут
- **Состояние per project** уже в схеме: таблица `sync_statuses` (поля `lastSyncAt`, `lastCommitSha`, `lastMrIid`, `status`, `errorMessage`). Запись создаётся при `connectProject` со `status='idle'`.
- **Колонка `projects.default_branch`** — добавить миграцией (требуется для шага 1 пайплайна; пока не сохраняется при подключении). При добавлении — обновить `connectProject` чтобы писать `remote.default_branch` в новое поле.
- **Идемпотентность**: вставка через `ON CONFLICT (gitlab_id) DO UPDATE` для commits/mrs/notes/tags
- **Пайплайн на один проект** (последовательно, чтобы не превышать rate limit):
  1. `fetchCommits` с `since=lastSyncAt` + `ref=defaultBranch` → upsert
  2. `fetchMergeRequests` с `updated_after=lastSyncAt` → upsert
  3. Для изменившихся MR: notes + approvals + changes → upsert + расчёт `firstReviewAt` / `approvedAt` / `linesAdded` / `linesRemoved`
  4. `fetchTags` → классификация в `deployments` по `release_tag_pattern`
  5. Запуск пересчёта `metrics_snapshots`
- **Защита от параллельного запуска** того же проекта: можно использовать `sync_statuses.status='running'` как мягкий лок + advisory lock в Postgres
- **Логирование sync**: каждый запуск пишет запись с длительностью и счётчиками (используется таблица `audit_logs` или отдельная `sync_runs`)
- **Endpoint manual trigger**: уже есть `POST /admin/projects/:uid/resync` (модуль projects делегирует в Sync) — реализация заработает после готовности этого пункта без правок в API
- **Endpoint статуса**: уже частично есть — `GET /admin/projects/:uid` возвращает `syncStatus`. Можно добавить отдельный `GET /admin/sync/status` с глобальным дашбордом.

### 1.4. Классификация деплоев и инцидентов (FR-03) — 🔴 5%

Сейчас только клиентский `fetchTags`. Логики mapping → `deployments` нет.

**Доработать:**

- Glob-матчер `releaseTagPattern` (`v*`, `release-*`, `**/v[0-9]*` и т.п.) — использовать `minimatch`
- Маппинг tag → `deployments`: `deployedAt = tag.commit.committed_date`, `tagName`, `environment='production'`
- Классификация хотфикса: MR с label из `projects.hotfixLabels` (по умолчанию `hotfix`, `rollback`, `revert`)
- Связь deploy ↔ предшествующий hotfix-MR — нужно для CFR
- Endpoint администратора: `PATCH /admin/projects/:uid/hotfix-labels` (массив строк)

## 2. Метрики (FR-04 и далее)

### 2.1. Cycle Time MR с декомпозицией — 🟡 15%

`cycle-time-mr.calculator.ts` — 21 строка, скелет. Данные в БД должны быть после sync.

**Доработать:**

- Реализовать формулы:
  - `timeToFirstReview = firstReviewAt − createdAt`
  - `timeInReview = approvedAt − firstReviewAt`
  - `timeToMergeAfterApproval = mergedAt − approvedAt`
- Агрегаты: медиана, p90 по выборке MR за период
- Игнорировать draft/WIP MR (поле или префикс title)
- Учёт пустых значений (MR без ревью → `timeToFirstReview = null`, не 0)

### 2.2. MR Size — 🟡 15%

`mr-size.calculator.ts` — 19 строк. Клиент-данные уже считаются (`computeMrSize`).

**Доработать:**

- Бакеты `≤50 / 51–200 / 201–400 / 401–800 / >800`
- Распределение по бакетам как `Record<bucket, count>` за период
- Медиана и p90 строк изменения

### 2.3. Lead Time for Changes — 🟡 15%

**Доработать:**

- Для каждого deployment найти first commit ветки: либо через `mr.firstCommitAt` (источник — `fetchMergeRequestCommits[0].authored_date`), либо через предыдущий deploy на том же ref
- `leadTime = deployedAt − firstCommitAt`
- Медиана и p90

### 2.4. Deployment Frequency — 🟡 15%

**Доработать:**

- Количество деплоев / период
- Категоризация Elite/High/Medium/Low (пороги из ВКР концепции)
- Группировка по дню/неделе/месяцу для графика

### 2.5. Change Failure Rate — 🟡 15%

**Доработать:**

- За окно после deploy (например, 7 дней) найти merged MR с hotfix-label
- `CFR = deploys_with_hotfix / total_deploys`
- Парная отрисовка с DF (обязательное требование ВКР)

### 2.6. Bus Factor — 🟡 15%

`bus-factor.calculator.ts` — 21 строка. `code-modules.service.ts` — 29 строк.

**Доработать:**

- Определение «модуля»: первая директория из пути файла (`src/auth/...` → `auth`). Источник — `mr.changes.new_path`. Хранить в таблице `code_modules` (есть в скелете).
- Расчёт: `count(distinct author_uid)` за последние 90 дней по каждому модулю
- Цветовая маркировка `<2 — red`, `2 — yellow`, `≥3 — green`

### 2.7. Snapshot-система — 🔴 5%

**Доработать:**

- `metrics_snapshots` уже в схеме, но writer отсутствует
- Cron-пересчёт после каждого sync: для каждой команды + каждой метрики + текущего периода писать snapshot
- TTL/чистка старых snapshots (или хранить навсегда для исторических графиков)
